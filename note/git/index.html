# 1.认识版本控制（版本控制）

什么是版本控制

  1. 版本控制的英文是Version control
  2. 是维护工程蓝图的标准作法，能追踪工程蓝图从诞生一直到定案的过程
  3. 版本控制也是一种软件工程技巧，借此能在软件开发的过程中，确保由不同人所编辑的同一程序文件都得到同步
  4. 简单来说，版本控制在软件开发中，可以帮助程序员进行代码的追踪、维护、控制等等一系列的操作

# 2.版本控制的功能

对于我们日常开发，我们常常面临如下一些问题，通过版本控制可以很好的解决：

  1. 不同版本的存储管理： 一个项目会不断进行版本的迭代，来修复之前的一些问题、增加新的功能、需求，甚至包括项目的重构；如果我们通过手动来维护一系列的项目备份，简直是一场噩梦
  2. 重大版本的备份维护： 对于很多重大的版本，我们会进行备份管理
  3. 恢复之前的项目版本： 当我们开发过程中发生一些严重的问题时，想要恢复之前的操作或者回到之前某个版本
  4. 记录项目的点点滴滴： 如果我们每一个功能的修改、bug的修复、新的需求更改都需要记录下来，版本控制可以很好的解决
  5. 多人开发的代码合并： 项目中通常都是多人开发，将多人代码进行合并，并且在出现冲突时更好的进行处理

# 3.版本控制的历史

版本控制的史前时代（没有版本控制）：人们通常通过文件备份的方式来进行管理，再通过diff命令来对比两个文件的差异；

CVS（Concurrent Versions System）

第一个被大规模使用的版本控制工具，诞生于1985年；由荷兰阿姆斯特丹VU大学的Dick Grune教授实现的，也算是SVN的前身（SVN的出现就是为了取代CVS的）。

SVN（Subversion）

  1. 因其命令行工具名为svn因此通常被简称为SVN
  2. SVN由CollabNet公司于2000年资助并发起开发，目的是取代CVS，对CVS进行了很多的优化
  3. SVN和CVS一样，也属于集中式版本控制工具
  4. SVN在早期公司开发中使用率非常高，但是目前已经被Git取代

Git（Linus的作品）

  1. 早期的时候，Linux社区使用的是BitKeeper来进行版本控制
  2. 但是因为一些原因，BitKeeper想要收回对Linux社区的免费授权
  3. 于是Linus用了大概一周的时间，开发了Git用来取代BitKeeper
  4. Linus完成了Git的核心设计，在之后Linus功成身退，将Git交由另外一个Git的主要贡献者Junio C Hamano来维护

# 4.集中式版本控制

CVS和SVN都是是属于集中式版本控制系统（Centralized Version Control Systems，简称 CVCS）

它们的主要特点是：

  1. 单一的集中管理的服务器，保存所有文件的修订版本
  2. 协同开发人员通过客户端连接到这台服务器，取出最新的文件或者提交更新

这种做法带来了许多好处，特别是相较于老式的本地管理来说，每个人都可以在一定程度上看到项目中的其他人正在做些什么。

但是集中式版本控制也有一个核心的问题：中央服务器不能出现故障：

  1. 如果宕机一小时，那么在这一小时内，谁都无法提交更新，也就无法协同工作
  2. 如果中心数据库所在的磁盘发生损坏，又没有做恰当备份，毫无疑问你将丢失所有数据

# 5.分布式版本控制

Git是属于分布式版本控制系统（Distributed Version Control System，简称 DVCS）

  1. 客户端并不只提取最新版本的文件快照， 而是把代码仓库完整地镜像下来，包括完整的历史记录
  2. 这么一来，任何一处协同工作用的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复
  3. 因为每一次的克隆操作，实际上都是一次对代码仓库的完整备份

电脑上要想使用Git，我们需要先对Git进行安装：

Git的官网：https://git-scm.com/downloads

根据自己的操作系统下载Git即可；

# 6.Bash – CMD – GUI 区别

Bash，Unix shell 的一种，Linux 与 Mac OS X 都将它作为默认 shell

  1. Git Bash 就是一个 shell，是 Windows 下的命令行工具，可以执行 Linux 命令
  2. Git Bash 是基于 CMD 的，在 CMD 的基础上增添一些新的命令与功能
  3. 所以建议在使用的时候，用 Bash 更加方便

Git CMD

  1. 命令行提示符（CMD）是 Windows 操作系统上的命令行解释程序
  2. 当你在Windows 上安装 git 并且习惯使用命令行时，可以使用 cmd 来运行 git 命令

Git GUI

  1. 基本上针对那些不喜欢黑屏（即命令行）编码的人
  2. 它提供了一个图形用户界面来运行 git 命令

# 7.Git的配置分类

既然已经在系统上安装了Git，你会需要做几件事来定制你的Git环境：

  1. 每台计算机上只需要配置一次，程序升级时会保留配置信息
  2. 你可以在任何时候再次通过运行命令来修改它们

Git 自带一个 git config的工具来帮助设置控制Git外观和行为的配置变量：

/etc/gitconfig 文件：包含系统上每一个用户及他们仓库的通用配置

  1. 如果在执行 git config时带上——system 选项，那么它就会读写该文件中的配置变量；

  2. 由于它是系统配置文件，因此你需要管理员或超级用户权限来修改它。（开发中通常不修改）

~/.gitconfig 或C/用户/用户名/.gitconfig 文件：只针对当前用户

  • 你可以传递——global 选项让Git读写此文件，这会对你系统上所有的仓库生效

当前使用仓库的Git 目录中的 config 文件（即.git／config）：针对该仓库

  • 你可以传递——local 选项让 Git强制读写此文件，虽然默认情况下用的就是它

# 8.Git的配置选项

安装Git后，要做的第一件事就是设置你的用户名和邮件地址。

  1. 这一点很重要，因为每一个 Git 提交都会使用这些信息，它们会写入到你的每一次提交中，不可更改
  2. 如果使用了 --global 选项，那么该命令只需要运行一次，因为之后无论你在该系统上做任何事情， Git 都会使用那些信息

检测当前的配置信息：git config --list

# 9.Git的别名（alias）

Git 并不会在你输入部分命令时自动推断出你想要的命令。如果不想每次都输入完整的 Git 命令，可以通过 git config 文件来轻松地为每一个命令设置一个别名

# 10.获取Git仓库 – git init/git clone

我们需要一个Git来管理源代码，那么我们本地也需要有一个Git仓库。

通常有两种获取 Git 项目仓库的方式：

  1. 初始化一个Git仓库，并且可以将当前项目的文件都添加到Git仓库中（目前很多的脚手架在创建项目时都会默认创建一个Git仓库）
  2. 从其它服务器克隆（clone）一个已存在的 Git 仓库（第一天到公司通常我们需要做这个操作）

方式一：初始化Git仓库git init

  1. 该命令将创建一个名为 .git 的子目录，这个子目录含有你初始化的 Git 仓库中所有的必须文件，这些文件是 Git 仓库的核心
  2. 但是，在这个时候，我们仅仅是做了一个初始化的操作，你的项目里的文件还没有被跟踪

# 11.文件的状态划分

现在我们的电脑上已经有一个Git仓库：

  1. 在实际开发中，你需要将某些文件交由这个Git仓库来管理
  2. 并且我们之后会修改文件的内容，当达成某一个目标时，想要记录下来这次操作，就会将它提交到仓库中

那么我们需要对文件来划分不同的状态，以确定这个文件是否已经归于Git仓库的管理：

  1. 未跟踪：默认情况下，Git仓库下的文件也没有添加到Git仓库管理中，我们需要通过add命令来操作
  2. 已跟踪：添加到Git仓库管理的文件处于已跟踪状态，Git可以对其进行各种跟踪管理

已跟踪的文件又可以进行细分状态划分：

  1. staged：暂缓区中的文件状态
  2. Unmodified：commit命令，可以将staged中文件提交到Git仓库
  3. Modified：修改了某个文件后，会处于Modified状态

在工作时，你可以选择性地将这些修改过的文件放入暂存区

# 17.Git的校验和

Git 中所有的数据在存储前都计算校验和，然后以==校验和==来引用。

  1. Git 用以计算校验和的机制叫做 SHA-1 散列（hash，哈希）
  2. 这是一个由 40 个十六进制字符（0-9 和 a-f）组成的字符串，基于 Git 中文件的内容或目录结构计算出来

# 18.查看提交的历史 – git log

在提交了若干更新，又或者克隆了某个项目之后，有时候我们想要查看一下所有的历史提交记录。

这个时候我们可以使用git log命令：

  1. 不传入任何参数的默认情况下，git log 会按时间先后顺序列出所有的提交，最近的更新排在最上面
  2. 这个命令会列出每个提交的 SHA-1 校验和、作者的名字和电子邮件地址、提交时间以及提交说明

==git log==

==git log --pretty=oneline==

--pretty＝：使用其他格式显示历史提交信息

--pretty=oneline：一行显示，只显示哈希值和提交说明

==git log --pretty=oneline --graph==

--graph：显示ASCII图形表示的分支合并历史

# 19.版本回退 – git reset

如果想要进行版本回退，我们需要先知道目前处于哪一个版本：Git通过HEAD指针记录当前版本。

  1. HEAD 是当前分支引用的指针，它总是指向该分支上的最后一次提交
  2. 理解 HEAD 的最简方式，就是将它看做==该分支上的最后一次提交的快照==

我们可以通过HEAD来改变Git目前的版本指向：

  1. 上一个版本就是HEAD^，上上一个版本就是HEAD ^^
  2. 如果是上1000个版本，我们可以使用HEAD~1000
  3. 我们可以可以指定某一个commit id

git reset --hard HEAD^ git reset --hard HEAD~1000 git reset --hard 2d44982

# 20.什么是远程仓库？

什么是远程仓库（Remote Repository）呢？

  1. 目前我们的代码是保存在一个本地仓库中，也就意味着我们只是在进行本地操作
  2. 在真实开发中，我们通常是多人开发的，所以我们会将管理的代码共享到远程仓库中

那么如何创建一个远程仓库呢？

  1. 远程仓库通常是搭建在某一个服务器上的（当然本地也可以，但是本地很难共享）
  2. 所以我们需要在Git服务器上搭建一个远程仓库

目前我们有如下方式可以使用Git服务器：

  1. 使用第三方的Git服务器：比如GitHub、Gitee、Gitlab等等
  2. 在自己服务器搭建一个Git服务

# 21.远程仓库的验证

常见的远程仓库有哪些呢？目前比较流行使用的是三种：

  1. GitHub (opens new window)
  2. Gitee (opens new window)
  3. 自己搭建Gitlab

对于私有的仓库我们想要进行操作，远程仓库会对我们的身份进行验证：

  • 如果没有验证，任何人都可以随意操作仓库是一件非常危险的事情

目前Git服务器验证手段主要有两种：

  1. 方式一：基于HTTP的凭证存储（Credential Storage）
  2. 方式二：基于SSH的密钥

下面我们来具体讨论一下这两种方式的验证规则和过程

# 22.远程仓库的验证 – 凭证

因为本身HTTP协议是无状态的连接，所以每一个连接都需要用户名和密码：

  1. 如果每次都这样操作，那么会非常麻烦
  2. 幸运的是，Git 拥有一个凭证系统来处理这个事情

下面有一些 Git Crediential 的选项：

  1. 选项一：默认所有都不缓存。 每一次连接都会询问你的用户名和密码
  2. 选项二：“cache” 模式会将凭证存放在内存中一段时间。 密码永远不会被存储在磁盘中，并且在15分钟后从内存中清除
  3. 选项三：“store” 模式会将凭证用明文的形式存放在磁盘中，并且永不过期
  4. 选项四：如果你使用的是 Mac，Git 还有一种“osxkeychain” 模式，它会将凭证缓存到你系统用户的钥匙串中（加密的）
  5. 选项五：如果你使用的是 Windows，你可以安装一个叫做 “Git Credential Manager for Windows (opens new window)” 的辅助工具

# 23.远程仓库的验证 – SSH密钥

Secure Shell（安全外壳协议，简称SSH）是一种加密的网络传输协议，可在不安全的网络中为网络服务提供安全的传输环境。

SSH以非对称加密实现身份验证

  1. 例如其中一种方法是使用自动生成的公钥-私钥对来简单地加密网络连接，随后使用密码认证进行登录
  2. 另一种方法是人工生成一对公钥和私钥，通过生成的密钥进行认证，这样就可以在不输入密码的情况下登录
  3. 公钥需要放在待访问的电脑之中，而对应的私钥需要由用户自行保管

如果我们以SSH的方式访问Git仓库，那么就需要生产对应的公钥和私钥：

ssh-keygen -t ed25519 -C “your email"

ssh-keygen -t rsa -b 2048 -C “your email"

查看远程地址：比如我们之前从GitHub上clone下来的代码，它就是有自己的远程仓库的

git remote add gitlab http://152.136.185.210:7888/coderwhy/gitremotedemo.git

# 30.Git标签（tag） - 创建tag

对于重大的版本我们常常会打上一个标签，以表示它的重要性：

  1. Git 可以给仓库历史中的某一个提交打上标签

# 32.Git提交对象（Commit Object）

几乎所有的版本控制系统都以某种形式支持分支

  • 使用分支意味着你可以把你的工作从开发主线上分离开来，以免影响开发主线

在进行提交操作时，Git 会保存一个提交对象（commit object）：

  1. 该提交对象会包含一个指向暂存内容快照的指针
  2. 该提交对象还包含了作者的姓名和邮箱、提交时输入的信息以及指向它的父对象的指针
  3. 首次提交产生的提交对象没有父对象，普通提交操作产生的提交对象有一个父对象
  4. 而由多个分支合并产生的提交对象有多个父对象

# 33.Git master分支

Git 的分支，其实本质上仅仅是指向提交对象的可变指针。

  1. Git 的默认分支名字是 master，在多次提交操作之后，你其实已经有一个指向最后那个提交对象的 master 分支
  2. master 分支会在每次提交时自动移动

Git 的 master 分支并不是一个特殊分支。

  1. 它就跟其它分支完全没有区别
  2. 之所以几乎每一个仓库都有 master 分支，是因为 git init 命令默认创建它，并且大多数人都懒得去改动它

# 44.Git rebase用法

在 Git 中整合来自不同分支的修改主要有两种方法：merge 以及 rebase。

什么是rebase呢？

在上面的图例中，你可以提取在 C4 中引入的补丁和修改，然后在 C3 的基础上应用一次；在 Git 中，这种操作就叫做变基（rebase）；你可以使用 rebase 命令将提交到某一分支上的所有修改都移至另一分支上，就好像“重新播放”一样。

rebase这个单词如何理解呢？

  1. 我们可以将其理解成改变当前分支的base
  2. 比如在分支experiment上执行rebase master，那么可以改变experiment的base为master

# 45.rebase的原理

rebase是如何工作的呢？

  1. 它的原理是首先找到这两个分支（即当前分支 experiment、变基操作的目标基底分支 master）的最近共同祖先 C2
  2. 然后对比当前分支相对于该祖先的历次提交，提取相应的修改并存为临时文件
  3. 然后将当前分支指向目标基底 C3
  4. 最后以此将之前另存为临时文件的修改依序应用

我们可以再次执行master上的合并操作： git checkout master

git merge experiment

# 46.rebase和merge的选择

事实上，rebase和merge是对Git历史的不同处理方法：

  1. merge用于记录git的所有历史，那么分支的历史错综复杂，也全部记录下来
  2. rebase用于简化历史记录，将两个分支的历史简化，整个历史更加简洁

了解了rebase的底层原理，就可以根据自己的特定场景选择merge或者rebase。

注意：rebase有一条黄金法则：永远不要在主分支上使用rebase

  1. 如果在主分支上面使用rebase，会造成大量的提交历史在main分支中不同
  2. 而多人开发时，其他人依然在原来的主分支中，对于提交历史来说会有很大的变化
